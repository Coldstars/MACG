# Structure Playbook

## Purpose

This playbook helps Writer avoid repetitive structures and choose a structure that fits the content type.

Workflow files decide which structure is required or allowed.

## Structure Selection Rule

Before drafting, Writer should identify:

- User intent
- Content type
- Audience maturity
- Decision stage
- Required human gate
- Required output sections
- Whether a comparison, tutorial, or explanation structure is best

## 1. How-To Guide Structure

Best for:

- Tutorials
- Setup guides
- Implementation questions
- “How to...” topics

Core structure:

1. What this guide helps you do
2. Who this is for
3. Prerequisites
4. Step-by-step workflow
5. Common issues
6. Best practices
7. FAQ
8. Next step

## 2. Comparison Structure

Best for:

- A vs B
- Alternative pages
- Difference topics
- Tool evaluation content

Core structure:

1. Direct answer
2. Quick comparison table
3. Where option A fits
4. Where option B fits
5. Decision criteria
6. Use case examples
7. FAQ
8. Recommendation summary

## 3. Use Case Structure

Best for:

- Scenario-based articles
- Product use cases
- Industry workflows

Core structure:

1. User problem
2. Why the current workflow fails
3. What data or capability is needed
4. How the workflow works
5. Practical scenarios
6. Evaluation checklist
7. FAQ

## 4. Problem-Solution Structure

Best for:

- Pain-point content
- Conversion-focused education
- Product positioning

Core structure:

1. The problem
2. Why it matters
3. Why common solutions fall short
4. What a better workflow looks like
5. Key capabilities
6. Implementation considerations
7. Product fit, if relevant
8. FAQ

## 5. API Tutorial Structure

Best for:

- Developer articles
- API guides
- Technical onboarding

Core structure:

1. What the API does
2. When to use it
3. Key parameters
4. Example request
5. Example response
6. Error handling
7. Integration tips
8. FAQ

## 6. Buyer Decision Structure

Best for:

- Commercial investigation content
- Choosing a vendor
- Evaluation checklists

Core structure:

1. Direct recommendation frame
2. Buyer needs
3. Must-have features
4. Nice-to-have features
5. Red flags
6. Evaluation checklist
7. Product fit
8. FAQ

## 7. Landing Page Structure

Best for:

- Product pages
- Topic pages
- Service pages

Core structure:

1. Hero promise
2. Pain-point block
3. Core value propositions
4. Use cases
5. How it works
6. Proof or credibility
7. FAQ
8. CTA sections

## 8. Help Center Structure

Best for:

- Documentation
- Feature instructions
- Support answers

Core structure:

1. What this article explains
2. Before you start
3. Steps
4. Configuration details
5. Expected result
6. Troubleshooting
7. Related articles

## Anti-Patterns

Do not:

- Use “What is / Why / How / FAQ” for every article.
- Use comparison tables when there is nothing meaningful to compare.
- Force product mention into every section.
- Start every piece with a dictionary definition.
- Add sections only to increase word count.
